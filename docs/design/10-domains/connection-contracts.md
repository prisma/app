# Connection contracts — typed service-to-service interfaces

Today a service-to-service Connection (`http()` in [core-model.md](core-model.md))
is untyped: it hydrates to a URL-anchored `fetch` wrapper, and nothing checks that
the producer actually answers what the consumer expects. This is the design for
making that dependency **typed and checked** — a **Contract** at the boundary.

The scope here is the model and the type-level enforcement. The precise `Contract`
interface (§ The Contract interface) is the one part still being finalized.

## Contract, and Data Contract

A **Contract** is the declared interface of a dependency: a value, owned by neither
side, that both the consumer and the provider refer to. It is the same idea as a
Prisma **Data Contract**, pointed at a different kind of dependency:

- an HTTP/service dependency → a **Contract** (methods + their input/output shapes);
- a data dependency → a **Data Contract** (the data shape a consumer depends on +
  migration compatibility).

Both are "the interface at a dependency boundary," checked and enforced across the
same planes. A bare "Contract" here means the service kind; "Data Contract" stays
the qualified data one; the umbrella a dependency declares is its **interface**.

The core is **parametric over the Contract**: it holds contract *values*, carries
their *types* (TypeScript does compile-time compatibility), and calls their
`satisfies()` method (runtime compatibility). It never inspects the protocol or the
comparison mechanics. So protocols (RPC first; gRPC / WebSocket / GraphQL later),
comparison mechanics (nominal now, structural later), and the number of outputs are
all ecosystem-open without the core changing.

## The four planes a Contract lives in

- **Authoring** — wiring a provider into a consumer's slot type-checks: the
  provider's exposed contract must be assignable to the consumer's required one.
- **Load** — the graph confirms the wired provider satisfies the consumer at build
  time (via the contract's `satisfies()`).
- **Run** — the consumer's `load()` returns a typed client bound to the contract.
- **Implement** — the provider satisfies the contract (a generated server), or the
  contract is derived from the provider.

## A neutral contract, an adapter on each end

The Contract is the port; the code on either side is an adapter:

- **Provider** — a *server adapter* implements the contract. The framework generates
  the server from the contract; the author supplies typed handlers (§ Implementing).
- **Consumer** — a *client adapter* binds the contract; it is what `load()` returns.

Because both adapters are generated by the same target pack, the wire format between
them is entirely private to the pack — not a framework concern.

**The binding does not have to be a network hop.** Once the client is generated from
the contract, the adapter behind it is chosen by the hex wiring:

- a co-deployed remote provider → the **network** adapter (a URL client);
- a co-located provider → an **in-memory** adapter (call the handler directly);
- a test → a **mock**.

Consumer code — `auth.verify(input)` — is identical across all three. That single
idea covers test mocks, the dependency-inversion swap, local dev with no deploy, and
wrapping a legacy server. A Connection is a port; *where* the provider runs is a
wiring decision the consumer never sees.

## The authoring model

Four touch-points, shown on the storefront-auth example. The Contract is defined
once, as a standalone value both sides import.

```ts
// auth.contract.ts — the shared Contract value. Its identity is the Load-time key.
export const authContract = contract({
  verify: rpc({ input: type({ token: "string" }), output: type({ ok: "boolean" }) }),
})

// auth/service.ts — the provider DECLARES what it exposes. `expose` is a record of
// named output ports, each its own contract. This is the single source of truth.
export default compute({
  build: node({ entry: "server.js" }),
  expose: { rpc: authContract, adminRpc: adminContract },
})

// auth/server.ts — the provider IMPLEMENTS it. Pass the service in; TS reads its
// `expose` and forces an exhaustive, correctly-typed handler map. serve() generates
// the server and calls load() for the deps, passing them to each handler.
import service from "./service"
export default serve(service, {
  rpc:      { verify: async ({ token }, { db }) => ({ ok: await check(db, token) }) },
  adminRpc: { disableUser: async ({ id }, { db }) => ({ disabled: true }) },
})

// storefront/service.ts — the consumer REQUIRES a contract (rpc(contract), not http())
export default compute({
  deps: { auth: rpc(authContract) },
  build: nextjs({ entry: "server.js" }),
})

// storefront/page.tsx — the consumer USES the derived typed client
const { auth } = service.load()
await auth.verify({ token })          // input {token}, output {ok}, both typed

// the hex — the WIRING is where the provider is checked against the consumer.
// authRef carries auth's exposed ports; pick one for the slot.
hex("storefront-auth", (h) => {
  const authRef = h.provision("auth", authService)
  h.provision("storefront", storefront, { auth: authRef.rpc })   // TS: rpc port must satisfy the slot
})
```

Where each piece carries the type that makes the wiring check itself:

- **`rpc(contract)`** on the consumer's dependency carries the *required* contract and
  hydrates to `Client<C>` (`{ verify(input): Promise<output> }`).
- **`expose`** on the provider service carries the *exposed* contracts, keyed by
  output-port name.
- **`provision(id, service)`** returns a `ProvisionedRef` whose named members
  (`authRef.rpc`) each carry their port's contract.
- **`provision(id, consumer, wiring)`** types `wiring` so each slot demands a ref
  whose contract is assignable to the slot's required contract. **TypeScript
  assignability is the "exposed ⊇ required" check**: a provider may expose more
  methods than the consumer needs (width), and per method inputs are contravariant,
  outputs covariant.

This is the hexagonal shape made concrete: a service has typed **input ports** (its
deps, consumed via `load()`) and typed **output ports** (its `expose`, served via
`serve()`); the hex connects an output port to an input port.

### Implementing — why the handler cannot skip the contract

`serve(service, handlers)` derives the required handler map from the service's
`expose`, so an incomplete or mistyped implementation does not compile:

```ts
function serve<S extends ServiceNode>(service: S, handlers: Handlers<ExposeOf<S>>): Server

type Handlers<E> = {
  [Port in keyof E]: { [M in keyof E[Port]]: (input: In<E[Port][M]>, deps: Deps) => Promise<Out<E[Port][M]>> }
}
```

The contract lives once, on the definition; the implementation is *handed* that
definition and cannot compile unless it satisfies it. The import points
definition-ward only (`server.ts` → `service.ts`, never the reverse), so bundling
`service.ts` into the runtime wrapper never pulls in the entry's app code — the
acyclic bundle shape from core-model.md is preserved.

One honest limit: the type system forces `serve(...)` to be complete and correct, but
cannot force the entry to *call* `serve` at all. `serve` is strictly the easier path
(fill handlers, no transport), and a deploy-time probe (call each method against the
started server, check the shape) can close the gap at runtime if a hard guarantee is
ever wanted.

## Compatibility — compile-time structural, runtime nominal

Enforcement happens in three places:

1. **Authoring (TypeScript) — structural, and the primary check.** Assignability at
   the wiring site rejects an incompatible provider before anything runs.
2. **Load — the contract's own `satisfies()`.** The framework calls
   `providerPort.satisfies(consumerSlot)` and does not know the mechanism. The first
   implementation is **nominal**: same contract value (identity). This matches
   Java-style name/identity comparison — structurally-equivalent-but-distinct
   contracts do not match.
3. **Run (per call) — validate input/output against the contract's schemas.** Catches
   a provider that is typed-compatible but lies at runtime — a bug, drift, or a
   legacy server wrapped in a Service that TypeScript never checked.

Growing nominal into **structural** later is backward-compatible: runtime accepts
"same value **or** structurally compatible," which is purely additive — everything
that passes today still passes. Because the mechanism is encapsulated behind
`satisfies()`, the change never touches the framework.

The distributed case is the important extension: when the provider is a *separately
deployed* Service whose TypeScript the local build cannot see, compatibility is
checked by comparing the consumer's contract against the provider's **published spec**
(the contract → OpenAPI / JSON-RPC output). That is the *same engine* as the Data
Contract migration check — "a newly deployed provider must still satisfy every
existing consumer." Structural comparison earns its keep there; the in-build hex is
fully covered by layers 1–3 today.

## The Contract interface — how compatibility is actually checked

The framework holds a `Contract` generically as a protocol **brand** plus an
**opaque comparison type** `Cmp`, and a runtime `satisfies()` method:

```ts
interface Contract<Kind extends string, Cmp> {
  readonly kind: Kind                                    // "rpc" · "grpc" · … (the brand)
  readonly __cmp: Cmp                                    // opaque to the core
  satisfies(required: Contract<Kind, unknown>): boolean  // runtime mirror of the compile check
}
```

**The compile-time compatibility check is plain TypeScript assignability — one line
in the core.** `provision` requires each wired contract to be assignable to the
consumer's required slot, with `NoInfer` on the brand so the `kind` is checked
rather than co-inferred into a union:

```ts
declare function provision<Deps extends Record<string, Contract<any, any>>>(
  id: string, consumer: Consumer<Deps>, wiring: { [K in keyof Deps]: NoInfer<Deps[K]> },
): void
```

The core never inspects `Cmp`. Correctness comes from the **kind**, which builds
`Cmp` so plain assignability means the right thing. For RPC, `Cmp` is a **concrete
function map**: `rpc()` returns a concrete `(input: I) => Promise<O>`, so
`contract()`'s `Cmp` is `{ verify: (input) => Promise<output>, … }` — not a mapped
type over a generic schema `M`:

```ts
declare function rpc<I, O>(m: { input: Schema<I>; output: Schema<O> }): (input: I) => Promise<O>
declare function contract<Fns extends Record<string, (i: any) => Promise<any>>>(fns: Fns): Contract<"rpc", Fns>
type Client<C> = C extends Contract<any, infer Cmp> ? Cmp : never
```

Because `Cmp` is a concrete function map, TypeScript applies real function variance to
`provided extends required` — **contravariant input, covariant output**, plus width
(a provider may expose extra methods and richer outputs). A provider that requires an
input field the consumer never sends is rejected; a wrong-kind provider is rejected
by the brand.

**The trap this avoids** (found by the executable proof, not reasoned): do *not*
parameterise the contract by the schema map `M` and derive `Client<M>` as a mapped
type. Comparing two mapped-type instantiations relates the `M`s covariantly, which
silently drops input-contravariance and accepts incompatible providers. The
comparison type must be a concrete function map, materialised by the kind's builder.

Schemas are **Standard Schema** (arktype the canonical authoring library; any
Standard-Schema validator works, so an Effect provider and a plain consumer
interoperate over one contract). They carry the runtime validators and drive the
OpenAPI/JSON-RPC spec; the *type* the core compares is the function map.

The executable proof of all of this — every accept/reject case, the three layers, and
the trap — is [`contract-satisfaction.poc.ts`](contract-satisfaction.poc.ts),
compiled under `--strict`.

## Ownership

The Contract is a **standalone value**; where it lives is code organisation. The
framework owns the builder and the type, never the location:

- a third-party hex ships the Contract *with* the provider; alternative impls and
  mocks import it;
- a first-party app puts the Contract beside the Service; mocks import it;
- a dependency-inverted app puts the Contract with the consumer or a central package,
  and implementer hexes depend on it.

All three are "both sides import one Contract value," which is also why identity-based
Load compatibility is enough today.

## Decisions and extension points

- **RPC first.** RPC removes the HTTP semantic surface (methods, paths, status
  codes); a contract collapses to `method → { input, output }`, which is trivial to
  compare and to generate. Legacy HTTP is out of scope — wrap it in a Service that
  satisfies an RPC contract.
- **Errors are deferred** from the MVP contract (`{ input, output }` only). Adding an
  `error` schema later is backward-compatible (an optional field).
- **Wire format** is private to the target pack (it owns both adapters).
- **PDL as an authoring surface.** A Contract is a value today (Standard Schema in
  TypeScript). Later it can be authored in Prisma Definition Language and compiled to
  the same value — the Data Contract is the natural first PDL output, the RPC Contract
  a later one. PDL changes the ergonomics, not the runtime.
- **New protocol kinds** (gRPC, WebSocket, GraphQL) are ecosystem packages: each
  provides its own `contract()`/`serve()`/client-binding, and slots in because the
  framework only asks TypeScript "assignable?" and the contract "satisfies?".
